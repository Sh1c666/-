# TCP 性能与 TLS 握手进阶

知识库补充条目,聚焦"慢"与"HTTPS 报错"两类难定位故障的深层机理。

## TCP 吞吐随丢包崩塌(Mathis 公式)

- **现象**:链路只有微量丢包(1%~2%),但大文件传输/接口吞吐断崖式下降,小请求却正常。
- **根因**:TCP 拥塞控制下,稳态吞吐约正比于 `MSS/(RTT·√p)`,p 为丢包率。1% 丢包即可让吞吐掉到原来的 1/10。
- **如何确认**:`icmp_ping` 见 1%~5% 丢包;`traceroute` 找到丢包/高延迟跳;`http_probe` 大对象慢但 TCP RTT 正常。
- **处置**:修复链路丢包(换线路/查拥塞跳);必要时调大拥塞窗口或启用 BBR。

## TLS 握手失败(非证书问题)

- **现象**:证书有效,但部分客户端无法建立 HTTPS(老 Java、旧 curl、嵌入式设备)。
- **根因**:客户端与服务器 TLS 版本或加密套件(cipher)无交集;或服务器要求 SNI 而客户端未发送。
- **如何确认**:`tls_inspect` 看 `tls_version`/`cipher`;对比客户端支持的版本(如仅 TLS1.0)。
- **处置**:服务器侧放开兼容的 cipher/版本(或反之升级客户端);确保 SNI 正确发送。

## 高延迟但零丢包(缓冲膨胀 Bufferbloat)

- **现象**:带宽测速正常,但交互(SSH/游戏/语音)明显卡顿,延迟抖动大。
- **根因**:链路设备队列过大,突发流量时排队导致延迟飙升,虽不丢包但体验差。
- **如何确认**:ping 在负载下延迟显著升高(空载正常);traceroute 见某跳 RTT 抖动。
- **处置**:启用队列管理(fq_codel/CAKE);限制上行带宽略低于实际上限。

## DNS 解析慢(非失败)

- **现象**:能解析但首次访问慢,之后正常(TTL 内快)。
- **根因**:本机 DNS 上游响应慢,或 NXDOMAIN-Retry 退避。
- **如何确认**:`dns_lookup` 计时偏长;对比直查公共 DNS(如 223.5.5.5 / 119.29.29.29)。
- **处置**:更换更快的本机/上游 DNS;检查是否有错误的搜索域后缀导致多次重试。
